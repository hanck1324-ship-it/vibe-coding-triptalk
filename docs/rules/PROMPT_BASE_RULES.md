# 프롬프트 기본 룰 (Prompt Base Rules)

모든 프롬프트 파일이 반드시 따라야 하는 기본 규칙입니다.

## 📌 필수 구성 요소

### 1. 헤더 (Header)
```txt
아래의 조건을 모두 적용하여, 아래의 요구사항을 모두 구현할 것.
구현 결과를 체크리스트로 반환할 것.
```

### 2. 조건-커서룰 (Cursor Rules)
```txt
==============================================

조건-커서룰) 아래의 커서룰을 적용하여 작업하고, 이 작업이 끝나면 해당 rules 적용 결과를 체크리스트로 반환할 것.
            - @01-common.mdc
            - @02-wireframe.mdc (와이어프레임)
            - @03-ui.mdc (UI)
            - @04-func.mdc (기능)
```

### 3. 조건-파일경로 (File Paths)
```txt
==============================================

조건-파일경로) 구현될 TSX 파일경로: src/components/[컴포넌트]/index.tsx
조건-파일경로) 구현될 CSS 파일경로: src/components/[컴포넌트]/styles.module.css
조건-파일경로) 구현될 HOOK 파일경로: src/components/[컴포넌트]/hook.ts (기능 프롬프트만)
조건-파일경로) 구현될 GQL 파일경로: src/components/[컴포넌트]/queries.ts (기능 프롬프트만)
조건-파일경로) (참고) 공통 GQL 타입: src/commons/graphql/graphql.ts
```

## 🚫 금지 사항

### 1. 코드 예시 제공 금지
❌ **나쁜 예:**
```txt
구현 예시:
    const [state, setState] = useState("");
```

✅ **좋은 예:**
```txt
- useState를 사용하여 상태 관리
```

### 2. HTML/JSX 구조 예시 금지
❌ **나쁜 예:**
```txt
<div className={styles.wrapper}>
  <input />
</div>
```

✅ **좋은 예:**
```txt
- wrapper > input 계층 구조
```

### 3. 변수명/함수명 강제 금지
❌ **나쁜 예:**
```txt
- const onChangeWriter = (e) => { ... }
```

✅ **좋은 예:**
```txt
- 작성자 입력 핸들러 함수 구현
```

### 4. 구체적인 구현 로직 지시 금지
❌ **나쁜 예:**
```txt
- if (writer && password) { setIsActive(true); }
```

✅ **좋은 예:**
```txt
- 필수 입력값이 모두 입력되면 버튼 활성화
```

## ✅ 권장 사항

### 1. 단계별 프롬프트 순서
```
01. wireframe (구조)
  ↓
201. ui (디자인)
  ↓
301. func (기본 기능)
  ↓
401. func.[기능명] (추가 기능 - 필요시)
  ↓
501. func.[기능명] (추가 기능 - 필요시)
```

### 2. 와이어프레임 프롬프트 특징
- **목적:** 구조만 만들기
- **스타일:** 최소한 (배경색, 여백만)
- **레이아웃:** flexbox만 사용 (absolute 금지)
- **내용:** 플레이스홀더 텍스트
- **길이:** 40-60줄

### 3. UI 프롬프트 특징
- **목적:** 피그마 디자인 구현
- **스타일:** 정확한 색상, 폰트, 크기
- **이미지:** 실제 이미지 사용
- **라이브러리:** Image, Icon 등 컴포넌트 사용
- **길이:** 60-100줄

### 4. 기능 프롬프트 특징
- **목적:** API, 상태관리, 비즈니스 로직
- **구조:** GQL → Hook → UI 연동
- **에러:** 에러 처리 필수
- **최적화:** refetch, 캐싱, 메모이제이션 (오버엔지니어링 주의)
- **길이:** 80-120줄

#### 4-1. 메모이제이션 적용 기준 (⚠️ 오버엔지니어링 주의!)

**✅ useMemo 적용 권장:**
- 복잡한 배열 연산 (filter, map, reduce 등)
- 객체/배열 변환 함수 호출
- 연산 비용이 높은 계산

**✅ useCallback 적용 권장:**
- props로 전달되는 함수
- 사용자 입력 핸들러 (onChange, onClick 등)

**❌ 오버엔지니어링 (피해야 할 것):**
- 간단한 함수에 useCallback 적용
- 단순 조회/getter에 useMemo 적용
- 모든 함수를 무조건 메모이제이션
- props로 전달되지 않는 일반 함수

**💡 판단 기준:**
"이 최적화가 성능 이득보다 코드 복잡도를 더 높이는가?"

## 🎯 "무엇을(What)" vs "어떻게(How)"

### ✅ 프롬프트에 적을 것 (What)
- 어떤 기능을 만들지
- 어떤 데이터를 보여줄지
- 어떤 라이브러리를 사용할지
- 어떤 요구사항을 충족해야 하는지

### ❌ 프롬프트에 적지 말 것 (How)
- 변수명을 뭐로 할지
- 함수 구조를 어떻게 짤지
- if문을 어떻게 쓸지
- 구체적인 구현 방법
